================================================================================
Desire Privé — анализ как готового продукта
Дата: 2026-07-16
================================================================================

Ориентир: ~20k строк TS/TSX, ~17 SQL-патчей, Next 14 + Supabase,
фичи «социалка + dating + E2EE-чат + premium + админка».

Это уже не MVP-заготовка — working product shell с сильным визуалом и
плотной доменной логикой. Но до «готового бизнеса» ещё несколько слоёв.


--------------------------------------------------------------------------------
1. ЧТО ЭТО ЗА ПРОДУКТ СЕЙЧАС
--------------------------------------------------------------------------------

Позиционирование: закрытый adult/dating club с:
  - анкетами и свайпами
  - гостями / друзьями
  - форумом
  - E2EE-чатом
  - квотами free vs Premium
  - модерацией

Сильная идея: не «ещё один Tinder», а приватный клуб
(luxury UI, cameo brand, E2EE, невидимка, жёсткие дневные лимиты free).
Это отличает от массовых dating-приложений.

Стадия: feature-rich pre-launch / early private beta.
Много сценариев «уже есть», мало «проверено на реальном трафике и деньгах».

2. СИЛЬНЫЕ СТОРОНЫ
--------------------------------------------------------------------------------

Область                  | Оценка          | Почему
-------------------------|-----------------|------------------------------------
Визуальный продукт       | сильный         | Единый luxury glass/gold язык,
                         |                 | light/dark, PWA-зачатки, mobile shell
Ширина фич               | очень высокая   | Feed, people, swipe, chat E2EE,
                         |                 | guests, friends, albums, reports,
                         |                 | support, admin, quotas
Монетизация (дизайн)     | здравая         | Лимиты на фото/анкеты/лайки/чаты +
                         |                 | superlike + invisible — freemium-крючок
Модерация                | выше среднего   | Баны, репорты, удаление контента/
                         |                 | чатов/фото, support tickets
Бэкенд-модель            | солидная база   | RLS, RPC rate limits, heartbeat,
                         |                 | retention, индексы, патчи под фичи
Темп итераций            | высокий         | Invisible, live presence, albums
                         |                 | move, retention — боли продукта

Ключевые «взрослые» решения, которые уже есть:
  - квоты free/premium на сервере
  - невидимка + last_active_at (retention не ломается)
  - E2EE + backup ключей (UX хоть как-то продуман)
  - отчёты / admin queues
  - удаление неактивных (анти-мусор БД)


--------------------------------------------------------------------------------
3. СЛАБЫЕ МЕСТА (ПО ПРИОРИТЕТУ)
--------------------------------------------------------------------------------

### P0 — блокеры «как бизнеса»

1. Нет реальных платежей
   Premium — цены и UI, premium_until выдаётся вручную/админом.
   Без ЮKassa/Stripe/CloudPayments это демо монетизации, не продукт.

2. E2EE — частично закрыто (2026-07)
   Cloud recovery: passphrase-encrypted key blob in e2ee_key_backups + E2eeRecoveryGate.
   Новый браузер: ввод пароля восстановления, без silent re-key.
   Backup file optional. Метаданные (кто/когда/typing) сервер по-прежнему видит —
   full metadata privacy без mixnet/архитектуры недостижима; UX честно об этом.

3. Схема БД = patch hell
   schema.sql + 15+ patch-файлов. Риск: прод и код разъезжаются
   (как с invisible / last_active). Нет миграций (Flyway/Supabase migrations),
   нет CI «apply + typecheck».

4. Нет тестов
   package.json: lint + typecheck. Нет unit/e2e.
   При dating + money + ban/quota — регрессии неизбежны.


### P1 — продукт и рост

5. Слишком много поверхностей сразу
   Feed + swipe + people + guests + friends + wall + albums + support + admin.
   Для пользователя нет одного «зачем я сюда пришёл сегодня».
   Классический риск: всё по чуть-чуть, retention слабый.

6. Cold start dating
   Без плотности людей в городе свайпы/поиск пустые.
   Квоты free (2 анкеты, 10 лайков) правильные для монетизации,
   но на пустой базе убивают онбординг.

7. Realtime presence — усилено (2026-07)
   3 слоя: Broadcast pulse (мгновенно, без publication) +
   postgres_changes на profiles (опционально) + adaptive poll 5s/20s.
   Невидимка шлёт offline broadcast сразу. SQL: patch_profiles_realtime.sql
   всё ещё полезен, но не обязателен для «зелёного кружка» в открытом чате.

8. Type safety Supabase
   Клиент «as any», partial database.ts.
   Быстро пишется, дорого чинить баги в SQL/RLS.


### P2 — качество, ops, trust

9. Безопасность adult — частично закрыто (2026-07)
   /legal/* без логина: rules 18+, terms, privacy (152-ФЗ), offer.
   AgeGate на входе (кроме /legal), чекбоксы 18+ + terms на регистрации.
   Жалобы underage/nsfw уже в reportReasons. Нет: KYC/фото-верификации возраста.

10. Антиспам / антибот — пока rate limits only (не трогали).

11. Наблюдаемость — trackEvent уже есть; funnel table depends on SQL patch.
    Полный dashboard — later.

12. i18n / multi-device / notifications — web push later; E2EE cloud backup
    закрывает multi-device ключи частично.

13. Performance / scale — later (после real DAU).

14. Документация оператора — docs/OPERATOR.md (premium, cron, SQL, E2EE support).

5. КОНКУРЕНТНАЯ РАМКА
--------------------------------------------------------------------------------

Вы между:
  - Dating apps (Tinder/Badoo/Pure) — denser matching, payments, push
  - Private clubs / adult communities — brand, privacy, paywall
  - Encrypted messengers — Signal-level UX ключей

Сейчас продукт шире dating, уже мессенджера, уже клуба.
Это плюс для story, минус для focus.


--------------------------------------------------------------------------------
6. КУДА ДВИГАТЬСЯ — ТРИ СТРАТЕГИЧЕСКИХ ТРЕКА
--------------------------------------------------------------------------------

### Трек A — «Launch money» (рекомендуется как основной на 4–6 недель)
Сделать так, чтобы можно было принимать деньги и не стыдно звать людей.

  1. Реальная оплата Premium (ЮKassa / CloudPayments) → webhook → premium_until
  2. Онбординг: 3–5 шагов (фото, город, цель, 1 свайп, 1 сообщение)
  3. Legal: 18+, privacy, terms, contacts
  4. Email/phone verify + базовый anti-abuse
  5. Supabase migrations (единый путь apply) + checklist «прод = код»
  6. Один metric dashboard: DAU, likes, chats started, pay conversion

Цель: платящие + понятный core loop.


### Трек B — «Dating core» (если трафик adult dating)
Сузить внимание:
  - Swipe + mutual + chat
  - Guests + profile quality
  - Меньше форума на главном пути

Feed оставить как retention/content, не как вход.


### Трек C — «Trust & privacy club» (если бренд важнее scale)
  - Невидимка / приватные альбомы / E2EE recovery
  - Invite-only / city clubs
  - Верификация фото
  - Дорогой premium

Медленнее рост, выше LTV.

10. КАК ВЫБРАТЬ СЛЕДУЮЩИЙ ХОД
--------------------------------------------------------------------------------

Ответить на 3 вопроса:

1. Через 60 дней успех =
   (а) первые 50 платящих,
   (б) 500 MAU в одном городе,
   (в) стабильный private club 100 invite-only?

2. Основной loop пользователя:
   свайп → match → chat
   или
   форум → профиль → чат?

3. Готовы ли 4 недели не пилить «красивое», а только:
   pay + onboarding + migrations + push?

От ответов зависит, режем ли feed/swipe scope или идём в payments-first.

Трек: A money / B dating / C private club.
